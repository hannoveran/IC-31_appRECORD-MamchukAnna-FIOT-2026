## 2. Короткі теоретичні відомості

### 2.1 Безпека Node.js API
Для backend-застосунків важливо захищати HTTP-рівень, перевіряти вхідні дані та обмежувати надмірну кількість запитів. У проєкті вже реалізовано хешування паролів, токенний доступ і валідацію полів, а в межах цієї лабораторної роботи додано захисні HTTP-заголовки та rate limiting.

Helmet встановлює набір HTTP-заголовків, які зменшують ризики XSS, MIME sniffing та інших типових web-атак. Rate limiting обмежує кількість запитів від одного клієнта за певний проміжок часу, що допомагає зменшити ризик brute-force або простого перевантаження API.

### 2.2 Оптимізація API
Оптимізація REST API включає зменшення обсягу відповіді, скорочення кількості звернень до бази даних і використання індексів. Для списку заявок додано пагінацію через параметри `page` та `limit`, тому API може повертати не весь список одразу, а лише потрібну сторінку.

Також у Prisma-схему додано індекси для полів, які часто використовуються у фільтрах або сортуванні: `status`, `priority`, `createdAt`.

### 2.3 Кешування
Кешування зберігає результат запиту на короткий час, щоб повторний запит не звертався до бази даних. Для лабораторної роботи використано простий in-memory cache. Він не потребує окремого Redis-сервера, але добре демонструє сам принцип кешування.

У відповідях API використовується заголовок `x-cache`:
- `MISS` — дані отримано з бази та записано в кеш;
- `HIT` — дані повернено з кешу.

Після створення або зміни сутності відповідний кеш очищається.

### 2.4 Тестування API
Автоматизовані тести дозволяють швидко перевірити, що базова поведінка API не зламалася після змін. Для цього використано Vitest і Fastify `inject`, який дозволяє виконувати HTTP-запити до застосунку без запуску окремого сервера на порту.

## 4. Реалізація backend-частини

### 4.1 Helmet і rate limiting
У файлі `app.js` підключено middleware безпеки `helmet` та обмеження кількості запитів через `express-rate-limit`.

```
app.use(helmet());

const limiter = rateLimit({
  windowMs: 1 * 60 * 1000,
  max: 100,
  message: {
    message: 'Забагато запитів, спробуйте пізніше',
  },
});

app.use(limiter);
```

Також окремо реалізовано ліміт для авторизації:

```
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 10,
  message: 'Too many login attempts',
});

app.use('/api/auth/login', loginLimiter);
```

Після цього API автоматично повертає захисні HTTP-заголовки та обмежує надмірну кількість запитів.

### 4.2 Redis cache
Для кешування списку змагань використано Redis. Підключення реалізовано у файлі `config/redis.js`.

```
const redis = new Redis({
  host: '127.0.0.1',
  port: 6379,
});
```

Redis використовується для тимчасового збереження результатів запитів до бази даних та зменшення навантаження на сервер.

### 4.3 Кешування списку змагань
У контролері `competitionController.js` реалізовано кешування маршруту `GET /api/competitions`.

Спочатку система перевіряє наявність даних у Redis:

```
const cachedCompetitions = await redis.get('competitions');
```

Якщо кеш існує — повертаються кешовані дані:

```
if (cachedCompetitions) {
  return res.json(JSON.parse(cachedCompetitions));
}
```

Якщо кешу немає — дані отримуються з MS SQL Server та записуються в Redis:

```
await redis.set(
  'competitions',
  JSON.stringify(data),
  'EX',
  60,
);
```

Термін життя кешу становить 60 секунд.

### 4.4 JWT-автентифікація
Для захисту API використано JWT-токени. Після успішного входу користувач отримує `accessToken`.

```
const accessToken = jwt.sign(
  { id: user.id, email: user.email, role: user.role },
  JWT_SECRET,
  { expiresIn: '15m' },
);
```

Перевірка токена виконується через middleware `authMiddleware.js`.

```
const decoded = jwt.verify(token, JWT_SECRET);

req.user = decoded;
```

Захищені маршрути доступні лише авторизованим користувачам.

### 4.5 Валідація даних
Для перевірки коректності введених даних використано `express-validator`.

Приклад валідації реєстрації користувача:

```
body('email').isEmail().withMessage('Некоректний email'),

body('password')
  .isLength({ min: 6 })
  .withMessage('Пароль мінімум 6 символів'),
```

Якщо дані некоректні, API повертає помилку `400 Bad Request`.

### 4.6 Навантажувальне тестування
Для тестування продуктивності API використано `Artillery`.

Приклад конфігурації тесту:

```
config:
  target: 'http://localhost:3001'
  phases:
    - duration: 30
      arrivalRate: 20

scenarios:
  - name: "Get competitions"
    flow:
      - get:
          url: "/api/competitions"
```

Під час тестування перевірено:
- стабільність роботи сервера;
- швидкість відповіді API;
- роботу rate limiting;
- коректність кешування Redis.

## 5. Автоматизоване тестування

### 5.1 Перевірка API
У межах проєкту використано перевірку API через інструменти типу `Artillery` та ручні запити (Postman). Основна увага приділена:
- стабільності endpoint-ів;
- роботі rate limiting;
- коректності відповідей при навантаженні;
- валідації даних на рівні API.

Приклад перевірки відповіді сервера:

```
const response = await fetch("http://localhost:3001/api/competitions");

console.log(response.status); // 200 або 429 при перевантаженні
```

Також перевірено захисні механізми сервера (rate limit):

```
HTTP 200 → нормальна відповідь
HTTP 429 → перевищення ліміту запитів
```

### 5.2 Навантажувальне тестування (Artillery)
Для перевірки продуктивності API використано `Artillery`.

Файл конфігурації тесту:

```
config:
  target: "http://localhost:3001"
  phases:
    - duration: 30
      arrivalRate: 20

scenarios:
  - name: Get competitions
    flow:
      - get:
          url: "/api/competitions"
```

Результати тестування показали:
- стабільну роботу API під навантаженням;
- спрацювання rate limiting;
- середній час відповіді в межах 1–3 ms;
- відсутність критичних помилок (`vusers.failed = 0`).

## 6. Postman-колекція для перевірки API

Для перевірки функціоналу API використано Postman-колекцію.

Основні запити:
- `POST /api/auth/register` — реєстрація користувача;
- `POST /api/auth/login` — авторизація;
- `GET /api/auth/me` — отримання профілю;
- `GET /api/competitions` — список змагань;
- `POST /api/competitions` — створення змагання;
- `PUT /api/competitions/:id` — оновлення;
- `DELETE /api/competitions/:id` — видалення.

Особливу увагу приділено:
- перевірці JWT-захисту;
- роботі middleware авторизації;
- коректності CRUD-операцій.

Для демонстрації рекомендується виконувати запити послідовно: реєстрація → логін → отримання токена → доступ до захищених маршрутів.

## 9. Висновки

У межах лабораторної роботи backend-частину Figure Skating Scoring System розширено механізмами автентифікації, обмеження запитів, кешування та тестування продуктивності. Реалізовано JWT-автентифікацію з access token, а також захищені маршрути для роботи з користувачем і змаганнями.

Для захисту API від перевантаження використано rate limiting, що обмежує кількість запитів від одного клієнта. У разі перевищення ліміту сервер повертає статус `429 Too Many Requests`, що було підтверджено під час навантажувального тестування Artillery.

Для роботи з даними використано Sequelize ORM із підключенням до MSSQL. Реалізовано CRUD-операції для сутності `Competition`, включно зі створенням, отриманням, оновленням та видаленням записів. Також враховано каскадне видалення пов’язаних даних через SQL-запити.

Окремо проведено навантажувальне тестування за допомогою Artillery. Тести показали стабільну роботу API при великій кількості запитів, але також зафіксували появу статусу `429`, що підтверджує коректну роботу механізму rate limiting.
