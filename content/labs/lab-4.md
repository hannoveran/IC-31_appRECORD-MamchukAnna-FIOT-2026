## 2. Короткі теоретичні відомості

### 2.1 Логування у Node.js-застосунках
Логування використовується для фіксації важливих подій у серверному застосунку: HTTP-запитів, помилок, дій користувачів, часу відповіді та службових повідомлень. Для навчального проєкту це дає змогу швидко перевірити, які маршрути викликалися, з яким статусом завершився запит і чи виникали помилки під час роботи API.

У поточному проєкті використано два рівні логування. Fastify logger виводить HTTP-події в консоль під час розробки, а Winston записує структуровані події у файли `app.log` та `error.log`. Такий підхід відповідає ідеї Morgan/Winston із завдання лабораторної роботи, але адаптований до вже реалізованого backend на Fastify.

### 2.2 Завантаження файлів
Файли з браузера або Postman передаються на сервер у форматі `multipart/form-data`. Звичайний JSON-парсер не обробляє такий формат, тому для Fastify використано плагін `@fastify/multipart`. Він дає змогу читати файл як stream, перевіряти його тип і розмір, а потім зберігати на диск.

У межах роботи дозволено завантажувати файли форматів `jpg`, `png` та `pdf`. Для запобігання надто великим запитам встановлено обмеження розміру файлу.

### 2.3 Моніторинг продуктивності
Базовий моніторинг Node.js-застосунку можна реалізувати через вбудований об'єкт `process`. Він дозволяє отримати:
- `process.uptime()` — час роботи процесу;
- `process.memoryUsage()` — використання оперативної пам'яті;
- `process.cpuUsage()` — використання CPU процесом.

Окремо для кожного HTTP-запиту вимірюється час обробки. Це допомагає побачити, які запити виконуються швидко, а які можуть потребувати оптимізації.

## 4. Реалізація backend-частини

### 4.1 Підключення middleware логування
Основні підключення виконано у файлі `server/app.js`. До Express-застосунку додано Morgan для логування HTTP-запитів, а також кастомний middleware для вимірювання часу обробки кожного запиту.

```
app.use(express.json());
app.use(responseTimeMiddleware);
app.use(morgan('dev'));
```

Middleware фіксує тривалість виконання кожного запиту та записує її у лог через Winston разом із HTTP-методом і URL.

### 4.2 Файловий логер Winston
Для логування створено окремий модуль `server/utils/logger.js`, який записує події у файл `app.log` та виводить їх у консоль.

```
const logger = winston.createLogger({
  level: 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.printf(({ level, message, timestamp }) => {
      return `${timestamp} [${level.toUpperCase()}]: ${message}`;
    })
  ),
  transports: [
    new winston.transports.Console(),
    new winston.transports.File({ filename: 'app.log' }),
  ],
});
```

У разі помилок використовується глобальний error middleware, який додатково логує їх через Winston.

### 4.3 Моніторинг стану сервера
Реалізовано endpoint `GET /status`, який повертає базову інформацію про стан сервера.

```
res.json({
  success: true,
  uptime: process.uptime(),
  memoryUsage: process.memoryUsage(),
  timestamp: new Date().toISOString(),
});
```

Endpoint використовується для перевірки працездатності API та базового моніторингу ресурсів.

### 4.4 Завантаження одного файлу
Реалізовано маршрут `POST /upload`, який дозволяє завантажити один файл за допомогою Multer. Файл зберігається у директорію `uploads`.

```
router.post('/', upload.single('file'), uploadFile);
```

Після успішного завантаження API повертає інформацію про файл (ім’я та статус операції).

### 4.5 Завантаження кількох файлів
Реалізовано розширений маршрут `POST /upload`, який підтримує завантаження кількох файлів одночасно.

```
upload.array('files', 5)
```

Максимальна кількість файлів за один запит — 5. Після обробки повертається масив назв збережених файлів.

### 4.6 Валідація файлів
Додано перевірку типів файлів та обмеження розміру.

Дозволені типи: jpg, jpeg, png, pdf

```
const allowedTypes = /jpeg|jpg|png|pdf/;
```

Також встановлено обмеження розміру файлу — 5 MB. У разі невідповідності файл відхиляється, а клієнту повертається помилка.

### 4.7 PM2-конфігурація
Для запуску застосунку у production-режимі використовується PM2. Сервер запускається як окремий процес із можливістю автоматичного перезапуску.

```
pm2 start app.js --name scoring-api
```

PM2 дозволяє переглядати логи, контролювати процеси та забезпечує стабільну роботу сервера.

## 5. Postman-колекція для перевірки

Для перевірки функціоналу backend-частини створено Postman-колекцію, яка містить основні сценарії тестування API вебзастосунку для оцінювання виступів з фігурного катання.

Колекція включає запити:
- `GET /` — перевірка роботи сервера;
- `GET /status` — моніторинг стану сервера (uptime, memory usage);
- `POST /upload` — завантаження одного файлу;
- `POST /upload` (multiple files) — завантаження кількох файлів;
- `POST /upload` (invalid file type) — перевірка валідації типу файлу;
- `POST /upload` (empty request) — перевірка обробки помилок при відсутності файлу.

Для тестування завантаження файлів використовується `form-data`. Успішні запити виконуються з файлами типу `.jpg`, `.png`, `.pdf`. Для перевірки валідації використовується файл з іншим розширенням (наприклад `.txt`).

## 6. Команди для запуску

### 6.1 Встановлення залежностей
```
npm install
```

### 6.2 Запуск API у режимі розробки
```
npm run dev
```

Після запуску API доступний за адресою: `http://localhost:3001`.

### 6.3 Перевірка роботи endpoint-ів
Основні перевірки виконуються через Postman або браузер:
- `GET /` — перевірка доступності сервера
- `GET /status` — стан сервера
- `POST /upload` — завантаження файлів

### 6.4 Перегляд логів Winston
```
type app.log
```

### 6.5 Запуск через PM2
```
pm2 start app.js --name scoring-api
pm2 list
pm2 logs scoring-api
pm2 restart scoring-api
pm2 stop scoring-api
```
